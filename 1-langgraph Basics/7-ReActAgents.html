ReAct Agent Architecture#

Aim#

This is the intuition behind ReAct, a general agent architecture.

  1. act - let the model call specific tools

  2. observe - pass the tool output back to the model

  3. reason - let the model reason about the tool output to decide what to do next (e.g., call another tool or just respond directly)

## State Schema
from typing_extensions import TypedDict
from langchain_core.messages import AnyMessage
from typing import Annotated
from langgraph.graph.message import add_messages
class State(TypedDict):
    messages:Annotated[list[AnyMessage],add_messages]

### Entire Chatbot With LangGraph
from IPython.display import Image, display
from langgraph.graph import StateGraph, START, END
from langgraph.prebuilt import ToolNode
from langgraph.prebuilt import tools_condition

### Node definition
def tool_calling_llm(state:State):
    return {"messages":[llm_with_tools.invoke(state["messages"])]}

# Build graph
builder = StateGraph(State)
builder.add_node("tool_calling_llm", tool_calling_llm)
builder.add_node("tools", ToolNode(tools))

builder.add_edge(START, "tool_calling_llm")
builder.add_conditional_edges(
    "tool_calling_llm",
    # If the latest message (result) from assistant is a tool call -> tools_condition routes to tools
    # If the latest message (result) from assistant is a not a tool call -> tools_condition routes to END
    tools_condition,
)
builder.add_edge("tools","tool_calling_llm")


graph = builder.compile()

# View
display(Image(graph.get_graph().draw_mermaid_png()))

What is machine learning
================================== Ai Message ==================================

**Machine learning (ML)** is a sub‑field of artificial intelligence that focuses on building systems that can **learn from data** and improve their performance on a given task without being explicitly programmed for every possible scenario.

### Core Idea
- **Data → Model → Prediction/Decision**  
  1. **Data** (examples, measurements, observations) are collected.  
  2. A **model** (a mathematical or computational structure) is chosen and its parameters are **trained** on the data, typically by minimizing a loss (error) function.  
  3. Once trained, the model can **generalize**: it makes predictions or decisions on new, unseen data.

### How It Differs from Traditional Programming
| Traditional Programming | Machine Learning |
|------------------------|------------------|
| **Explicit rules** are written by a programmer. | **Implicit rules** are inferred from data. |
| The program’s behavior is fixed unless the code changes. | The program’s behavior changes as it sees more data (re‑training, online learning). |
| Works well for tasks with clear, deterministic logic. | Excels at tasks where patterns are complex, noisy, or not well understood (e.g., image recognition, language translation). |

### Main Categories

| Category | Typical Goal | Example Algorithms |
|----------|--------------|--------------------|
| **Supervised learning** | Learn a mapping from inputs **x** to known outputs **y** (labels). | Linear regression, logistic regression, support‑vector machines, decision trees, random forests, neural networks. |
| **Unsupervised learning** | Discover structure in data without labeled outcomes. | K‑means clustering, hierarchical clustering, principal component analysis (PCA), autoencoders, generative adversarial networks (GANs). |
| **Semi‑supervised learning** | Combine a small amount of labeled data with a large amount of unlabeled data. | Self‑training, graph‑based methods, co‑training. |
| **Reinforcement learning** | Learn a policy for sequential decision‑making by interacting with an environment and receiving rewards. | Q‑learning, Deep Q‑Network (DQN), Proximal Policy Optimization (PPO). |
| **Self‑supervised learning** | Create surrogate labels from the raw data itself, enabling large‑scale pre‑training. | Masked language modeling (BERT), contrastive learning (SimCLR). |

### Typical Workflow

1. **Problem definition** – Clarify the task (classification, regression, clustering, etc.) and success metrics (accuracy, F1‑score, mean‑squared error, etc.).
2. **Data collection & preprocessing** – Gather raw data, clean it, handle missing values, normalize/scale, encode categorical variables, split into training/validation/test sets.
3. **Model selection** – Choose an algorithm family and a specific architecture/hyper‑parameters.
4. **Training** – Optimize the model’s parameters on the training set (often via gradient‑based methods like stochastic gradient descent).
5. **Evaluation** – Measure performance on validation data; tune hyper‑parameters; guard against over‑fitting (e.g., using cross‑validation, regularization, early stopping).
6. **Deployment** – Export the trained model to production (as a web service, embedded device, etc.).
7. **Monitoring & maintenance** – Track real‑world performance; retrain when data distribution shifts (concept drift).

### Common Applications

- **Computer vision** – Object detection, facial recognition, medical imaging analysis.
- **Natural language processing** – Sentiment analysis, machine translation, chatbots, summarization.
- **Recommendation systems** – Personalized product, movie, or music suggestions.
- **Finance** – Fraud detection, algorithmic trading, credit scoring.
- **Healthcare** – Predictive diagnostics, drug discovery, patient risk stratification.
- **Autonomous systems** – Self‑driving cars, robotics, drones.
- **Industrial** – Predictive maintenance, quality control, demand forecasting.

### Key Concepts & Terminology

- **Feature** – A measurable property or attribute used as input to the model.
- **Label** – The ground‑truth output the model is trained to predict (only in supervised learning).
- **Loss / Cost function** – Quantifies the error between predictions and true labels; the training process minimizes this.
- **Gradient descent** – An iterative optimization algorithm that updates parameters in the direction that reduces loss.
- **Overfitting / Underfitting** – Overfitting: model captures noise, performs poorly on new data. Underfitting: model is too simple to capture underlying patterns.
- **Regularization** – Techniques (L1/L2 penalties, dropout, early stopping) that discourage overfitting.
- **Cross‑validation** – Splitting data into multiple train/validation folds to obtain a more reliable estimate of model performance.
- **Bias‑variance trade‑off** – Balancing systematic error (bias) against sensitivity to data fluctuations (variance).

### Popular Tools & Libraries

- **Python**: scikit‑learn, TensorFlow, PyTorch, Keras, XGBoost, LightGBM, CatBoost.
- **R**: caret, mlr, tidymodels.
- **Cloud platforms**: AWS SageMaker, Google Cloud AI Platform, Azure Machine Learning.
- **MLOps frameworks**: MLflow, Kubeflow, DVC, TensorBoard.

### Summary
Machine learning provides a **data‑driven** way to create predictive or decision‑making systems. By automatically extracting patterns from large datasets, ML enables solutions to problems that are too complex for hand‑crafted rules, powering many of today’s intelligent applications across virtually every industry.

MemorySaver#

LangGraph can use a checkpointer to automatically save the graph state after each step.

This built-in persistence layer gives us memory, allowing LangGraph to pick up from the last state update.

One of the easiest checkpointers to use is the MemorySaver, an in-memory key-value store for Graph state.

All we need to do is simply compile the graph with a checkpointer, and our graph has memory!

from langchain_openai import ChatOpenAI
llm = ChatOpenAI(model="gpt-4o")
llm_with_tools = llm.bind_tools(tools)

### Entire Chatbot With LangGraph
from IPython.display import Image, display
from langgraph.graph import StateGraph, START, END
from langgraph.prebuilt import ToolNode
from langgraph.prebuilt import tools_condition

### Node definition
def tool_calling_llm(state:State):
    return {"messages":[llm_with_tools.invoke(state["messages"])]}

# Build graph
builder = StateGraph(State)
builder.add_node("tool_calling_llm", tool_calling_llm)
builder.add_node("tools", ToolNode(tools))

builder.add_edge(START, "tool_calling_llm")
builder.add_conditional_edges(
    "tool_calling_llm",
    # If the latest message (result) from assistant is a tool call -> tools_condition routes to tools
    # If the latest message (result) from assistant is a not a tool call -> tools_condition routes to END
    tools_condition,
)
builder.add_edge("tools","tool_calling_llm")

from langgraph.checkpoint.memory import MemorySaver
memory=MemorySaver()
graph_memory = builder.compile(checkpointer=memory)
